# Radio Cloud Audio Editor & FFmpeg Stream

This project is a full-stack radio automation system consisting of a Node.js backend for audio streaming and playlist management, and a React frontend for visualization and control.

## Project Structure

The project is divided into two main directories:

*   **`ffmpeg-backend/`**: The server-side application.
*   **`frontend-ffmpeg-playlist/`**: The client-side user interface.

## Backend (`ffmpeg-backend/`)

The backend is built with **Node.js**, **Express**, and **Socket.IO**. It uses `fluent-ffmpeg` to mix and stream audio.

### Key Components

*   **`server.mjs`**: The application entry point. Initializes the Express app, HTTP server, and Socket.IO. Starts the scheduler and radio engine.
*   **`services/radio.js`**: The core radio engine.
    *   Manages FFmpeg processes for audio playback and mixing.
    *   Handles the "Live-Safe" playlist editing logic: mutations (drag-and-drop) are persisted to a new JSON file (Copy-on-Write) and immediately reloaded to ensure the active state matches the file system without interrupting the stream.
    *   Implements ID-based playback (`playTrackById`) to support seamless reordering.
*   **`services/playlist.js`**: Handles file I/O for playlist JSONs.
    *   `loadPlaylist(specificPath)`: Loads playlist data from disk. Can accept a specific path to force-load edits.
    *   `saveEditedPlaylist(tracks, time)`: Persists modified playlists to `..._edited-sanoma-playlist.json` files to maintain state across restarts/reloads.
    *   `parseJsonToTracks`: parses the proprietary JSON format into a standardized track object, assigning UUIDs for stable identification.
*   **`services/controlSocket.js`**: Manages WebSocket communication.
    *   Emits events: `state`, `status_update`, `playlist_update`, `radio_channel_info`.
    *   Handles commands: `start`, `stop`, `next`, `playlist:insert`, `player:play`.
*   **`store.js`**: Centralized in-memory state management (`state` object).

### Data Flow

1.  **Playlist Loading:** The system scans `song_data/` for JSON files matching the current hour/date. Edited files (`*_edited-sanoma-playlist.json`) take precedence.
2.  **Playback:** The radio engine (`radio.js`) reads audio files from `songs/`, processes them with FFmpeg (fades/mix points), and streams the output to connected clients.
3.  **Editing:** When a user modifies the playlist (drag-and-drop):
    *   Frontend emits `playlist:insert`.
    *   Backend creates a copy of the playlist, modifies it, and saves it to a new JSON file.
    *   Backend *immediately* reloads the state from this new file to ensure synchronization.
    *   Backend broadcasts `playlist:updated` to all clients.

## Frontend (`frontend-ffmpeg-playlist/`)

The frontend is a **React** application built with **Vite** and styled with **Tailwind CSS**.

### Key Components

*   **`src/components/playlist/PlaylistViewer.jsx`**: The main view.
    *   Connects to the backend via Socket.IO.
    *   Listens for `playlist_update` and `status_update`/`player:state` events to render the UI.
    *   **Crucial:** It does *not* optimistically update local state for complex operations. It waits for the authoritative state from the backend to ensure consistency.
    *   Handles drag-and-drop actions (`playlist:insert`) and playback requests (`player:play`).
*   **`src/components/playlist/PlaylistTable.jsx`**: Renders the list of tracks. Implements HTML5 Drag and Drop API.
*   **`src/components/ControlPanel.jsx`**: Provides global controls (Start, Stop, Next, Auto/Manual mode).

## Building and Running

### Prerequisites

*   Node.js (v18+ recommended)
*   pnpm (or npm/yarn)
*   FFmpeg installed and available in the system PATH.

### Backend

```bash
cd ffmpeg-backend
pnpm install
node server.mjs
```

The server typically runs on port `3000` (check `config.js` or `.env`).

### Frontend

```bash
cd frontend-ffmpeg-playlist
pnpm install
pnpm dev
```

The frontend will run on a port determined by Vite (usually `5173`).

## Development Conventions

*   **State Management:** The backend is the single source of truth. The frontend is a projection of the backend state.
*   **Identification:** Tracks are identified by immutable `uuid`s generated by the backend, not by array index. This allows for safe reordering.
*   **Persistence:** Edits are not just in-memory; they are written to disk to survive scheduler syncs and restarts.
*   **Tooling:** Uses `eslint` for linting.
